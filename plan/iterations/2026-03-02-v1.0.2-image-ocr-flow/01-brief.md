# 01 Brief

版本
- 版本号：v1.0.2
- 目标日期：2026-03-04

一句话目标
- 打通“前端上传图片 -> 聊天记录展示图片 -> 后端落盘并 OCR -> 喂给大模型 -> 返回回复”的完整链路。

问题/背景
- 当前后端 OCR 能力存在，但图片消息链路不完整：前端上传入口未实现、历史消息未按图片类型展示。
- 目前图片媒体在后端没有稳定的本地 URL 存储策略，导致记录回放与资源管理不理想。
- 用户期望：图片在聊天中正常展示，后端保存图片并将 OCR 文本参与模型问答，数据库保存图片 URL。

范围
- 需要做：
- 前端实现图片上传入口和图片消息渲染，支持发送时 loading 态
- 后端实现图片落盘存储并返回可访问 URL
- 后端 OCR 识别结果拼接到发送给大模型的内容中
- 聊天记录落库 `type=image` 且 `media` 存储图片 URL
- 历史消息回放时正确显示图片
- 不做：
- 不改数据库表结构（复用 `chat_records.type/media`）
- 不改鉴权体系与会话接口协议
- 不处理图片 CDN/对象存储（先本地文件存储）

接口或数据变更
- `POST /api/ai/chat` 支持 multipart 图片上传时，后端将图片保存到本地并把图片 URL 写入 `chat_records.media`
- `GET /api/ai/sessions/:id/messages` 维持原结构，前端改为使用 `type/media` 渲染图片

风险与依赖
- OCR 识别耗时可能较长，前端必须保持明确 loading 反馈
- 本地文件路径与静态资源路由需统一，避免图片 URL 在不同环境失效
- 需兼容文本消息流程，避免影响现有 SSE 回复链路
